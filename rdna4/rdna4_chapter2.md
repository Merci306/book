# 第二章：着色器概念

RDNA4 着色器程序（内核）是由着色器处理器执行的程序。从概念上讲，着色器程序在每个工作项上独立执行，但实际上处理器将最多 32 或 64 个工作项组合成一个 wave，以一次 pass 在全部 32 或 64 个工作项上执行该着色器程序（"wave32" 或 "wave64"）。

RDNA4 处理器主要由以下部分组成：

- **标量 ALU**：对每个 wave 操作单个值（所有工作项共用）
- **向量 ALU**：对每个工作项操作独立的值
- **本地数据存储**：允许工作组内的工作项通信和共享数据
- **标量内存**：通过缓存将数据从内存传输到 SGPR
- **向量内存**：在 VGPR 与内存之间传输数据，包括纹理贴图采样
- **导出**：将数据从着色器传输到专用图形渲染硬件

程序控制流使用标量 ALU 指令处理，包括 if/else、分支和循环。标量 ALU（SALU）和内存指令作用于整个 wave，操作最多两个 SGPR 以及字面常量。

向量内存和 ALU 指令同时对 wave 中的所有工作项进行操作。为了支持分支和条件执行，每个 wave 都有一个 **EXECute 掩码**，用于确定哪些工作项当前处于活动状态，哪些处于休眠状态。活动工作项执行向量指令，休眠工作项将指令视为 NOP。EXEC 掩码可由标量 ALU 指令或向量 ALU 比较随时写入。

向量 ALU 指令通常最多可接受三个参数，这些参数可来自 VGPR、SGPR 或指令流中的字面常量。它们对 EXEC 掩码所启用的所有工作项进行操作。向量比较和带进位加法会将每工作项的位掩码返回到 SGPR，以指示哪些工作项的比较结果为"真"或产生了进位输出。

向量内存指令在 VGPR 与内存之间传输数据，每个工作项提供自己的内存地址并提供或接收唯一的数据。这些指令同样受 EXEC 掩码约束。

## 2.1 Wave32 与 Wave64

着色器支持 32 个工作项的 wave（"wave32"）和 64 个工作项的 wave（"wave64"）。

除 VOPD（双 VALU）外，所有操作都支持两种 wave 大小，但着色器程序必须针对特定 wave 大小编译和运行，无论任何给定 wave 中有多少工作项处于活动状态。

**指令发射规则：**

- **Wave32**：每条指令最多发射一次
- **Wave64**：每条指令通常执行两次——一次针对低半部分（工作项 31-0），然后再次针对高半部分（工作项 63-32）

这种两次执行仅适用于 VALU 和 VMEM（LDS、纹理、缓冲区、flat、暂存、全局）指令；标量 ALU 和内存以及分支和消息无论 wave 大小只发射一次。导出请求也只发射一次，不受 wave 大小影响。

如果某一半的 EXEC 掩码全为零，硬件可以选择跳过该半部分，但不会同时跳过 VMEM 指令的两个半部分，因为这会混淆未完成内存指令计数器（除非该 wave 没有未完成的 VMEM 指令）。对于写入 SGPR 的 VALU 指令，也不会跳过任何一半。有关指令跳过规则的详细信息，请参见 3.2.2 节。

**Wave64 的第二次 pass 差异：**

两次 pass 均使用指令执行前的 wave 状态；wave64 的第一次 pass 不影响第二次 pass 的输入。

> 注：对于既读又写同一 SGPR 的 wave64 VALU 指令，存在例外情况，请参见"Wave64 目标 SGPR 限制"。

除了 EXEC 掩码在低半和高半之间不同外，两次 pass 的标量输入也可能不同。两次 pass 使用相同的常量，但掩码和进位输入/输出不同。

第二次 pass 的差异：
- **输入递增**：进位输入、div-fmas 和 v_cndmask 均使用下一个 SGPR（SSRC + 1，或 VCC_HI）
- **输出递增**：进位输出、div-scale 和 v_cmp 均写入下一个 SGPR（SDST + 1，或 VCC_HI）
  - v_cmpx 写入 EXEC_HI 而不是 EXEC_LO

Wave32 wave 忽略 EXEC 和 VCC 的高 32 位。VCCZ 和 EXECZ 分别反映 wave32 wave 的 VCC 和 EXEC 最低 32 位的状态。

## 2.2 着色器类型

### 2.2.1 计算着色器

计算内核（着色器）是可在 RDNA4 处理器上运行的通用程序，从内存获取数据，处理后将结果写回内存。计算内核由一个调度（dispatch）创建，该调度使 RDNA4 处理器在 1D、2D 或 3D 数据网格的所有工作项上运行内核。RDNA4 处理器遍历该网格并生成 wave，然后运行计算内核。每个工作项以其在网格内的唯一地址（索引）初始化，基于此索引，工作项计算需要处理的数据地址以及如何处理结果。

### 2.2.2 图形着色器

着色器支持 3 种图形 wave 类型：像素着色器（PS）、几何着色器（GS）和曲面细分着色器（HS）。

**渲染模式（启动行为）：**

- **普通模式**：几何引擎（GE）向 wave 启动硬件发送信息，为每个元素（图元）启动的 wave 初始化 VGPR；GE 获取索引和顶点缓冲区数据并加载到 VGPR
- **网格着色器模式**：将 GS 启动转变为类似 CS 的启动，wave 启动硬件进行展开以生成元素并在飞行中生成元素索引。网格着色器程序决定如何使用该索引值。

放大着色器决定启动多少个网格着色器组。网格着色器先处理顶点，然后处理图元。

## 2.3 工作组

工作组是一组可以通过 LDS 共享数据并可在 barrier 处同步的 wave 集合。barrier 是一种同步原语，使每个 wave 到达着色器中的给定点后，任何 wave 才能继续执行。

工作组中的所有 wave 都发射到同一个 WGP，但可以运行在任意 4 个 SIMD32 上，并可通过 LDS 共享数据。WGP 最多支持 32 个工作组，每个工作组最多 1024 个工作项。

工作组中的 wave 最多可共享 64kB 的 LDS 空间分配。仅由单个 wave 组成的工作组不计入 32 个工作组的限制，它们不分配 barrier 资源，barrier 操作被视为 S_NOP。

每个工作组或 wave 可在以下两种模式之一中运行，在 wave 创建时通过每次绘制/调度进行选择：

### CU 模式

在此模式下，LDS 被有效地分为独立的上半部分和下半部分，分别服务于两个 SIMD32。Wave 被分配与其运行的 SIMD 关联的 LDS 半部分的空间。对于工作组，所有 wave 被分配到同一对 SIMD32。此模式由于两个半部分并行运行可能提供更快的操作，但限制了数据共享（上半部分的 wave 无法读取下半 LDS 中的数据，反之亦然）。在 CU 模式下，工作组的所有 wave 都驻留在同一个 CU 内。

### WGP 模式

在此模式下，LDS 是一块大型连续内存，WGP 上的 wave 从中分配，最大分配大小相同。在 WGP 模式下，工作组的 wave 可以分布在 WGP 中的两个 CU（全部 4 个 SIMD32）上。WGP 模式不支持 `DS_PARAM_LOAD` 和 `DS_DIRECT_LOAD`。

WGP（及 LDS）可以同时运行一些 WGP 模式的 wave 和其他 CU 模式的 wave。当 wave 引用与其所在 SIMD"对侧"的数据时，LDS 性能可能会降低。

## 2.4 着色器填充要求

由于 RDNA4 设备使用了激进的指令预取，用户必须在所有着色器末尾额外填充 **64 个 DWORD（256 字节）**的数据。建议使用 `S_CODE_END` 指令作为填充，以便在调试时更容易识别着色器的结束位置。这确保了指令预取硬件在超过着色器末尾时，不会访问到未初始化的内存（或未映射的内存页）。

---

## 2.5 Whole Quad Mode（整体四元组模式）

Whole Quad Mode（WQM）是一种操作方式：如果一组 4 个线程中的任何一个线程被启用，则所有 4 个线程都表现为已启用。这种模式对于依赖四元组中所有线程被启用来计算数据的操作非常有用，例如需要知道相邻像素值的操作。SAMPLE 细节层次（LOD）计算是一个常见示例。

Whole Quad Mode 可以使用 `S_WQM_B32` 或 `S_WQM_B64` 显式应用，通过修改 EXEC 掩码以 4 个 lane 为单位启用组。

以下情况会临时应用 Whole Quad Mode：

- **V_INTERP_\* 指令**：使用四元组中 3 个 lane 的数据
- **DS_PARAM_LOAD、DS_DIRECT_LOAD**：每个四元组将数据加载到 3 个 lane
- **SAMPLE**：使用相邻像素数据计算细节层次的操作
