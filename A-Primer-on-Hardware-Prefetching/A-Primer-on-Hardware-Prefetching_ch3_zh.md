# 第 3 章　数据预取（Data Prefetching）

**《硬件预取入门》（A Primer on Hardware Prefetching）**  
Babak Falsafi & Thomas F. Wenisch，2014

---

数据 miss 模式源于算法和高级编程结构在内存中组织与遍历数据的固有方式。与指令 miss 模式相比，数据访问模式**更加多样**——尤其是指针链表等数据结构，支持多种遍历方式；而且数据结构在执行过程中动态变化，导致遍历模式随之改变。这种复杂性使数据预取的设计空间远比指令预取更为丰富。

本章将数据预取器分为四大类：

| 类别 | 核心依据 |
|------|---------|
| **步长/流预取（Stride/Stream）** | 规则的步长访问模式 |
| **地址关联预取（Address-Correlating）** | 重复出现的 miss 地址序列（时序关联） |
| **空间关联预取（Spatially Correlated）** | 数据结构布局的规律性（空间关联） |
| **基于执行的预取（Execution-Based）** | 超前执行指令序列，直接发现地址计算 |

---

## 3.1 步长与流预取（Stride and Stream Prefetchers）

步长/流预取器是下一行指令预取的数据版本，适用于**连续或等步长布局**的数据，对密集矩阵和数组访问效果极佳，但对指针数据结构几乎无效。

### 核心机制

**Baer & Chen 的参考预测表（Reference Prediction Table）**：
- 以 load 指令的 PC 为索引键，每个表项记录该 load 的最近访问地址和相邻两次访问的地址差（步长）；
- 连续两次观察到相同步长时，用最近地址 + 步长计算出一个或多个预取地址；
- 持续匹配的步长访问序列称为一个**流（stream）**。

### 关键设计问题

1. **区分多个交错步长序列**（如矩阵-向量乘法中多个 load 的步长序列相互交织）；
2. **预取深度（prefetch degree/depth）**：太浅则延迟未被隐藏，太深则预取数据被替换或造成污染；Hur & Lin 提出自适应状态机，根据近期流长度直方图动态调整预取深度，对短流也有效。

### 流缓冲区（Stream Buffers）

Jouppi 提出将预取数据放入独立的**流缓冲区**（与 L1 并行访问），而非直接放入缓存：
- 低准确率的流不会替换缓存中的有用数据，降低污染风险；
- 缓冲区按 FIFO 或关联方式管理，命中时将块移入 L1 并预取流中下一块；
- 检测到新流时，按轮转或 LRU 策略清空并重分配流缓冲区。

## 3.2 地址关联预取（Address-Correlating Prefetchers）

步长预取对指针数据结构无效。地址关联预取器专门针对**指针追逐**访问模式，利用算法反复以相同方式遍历数据结构这一特点，从历史 miss 序列中发现**时序关联（temporal correlation）**——两个地址在时间上相邻地被访问，则将来往往也会相邻访问。

### 3.2.1 跳转指针（Jump Pointers）

最早期的方案：在链表节点中增加一个"跳 N 步"的指针，预取器沿跳转指针超前于主遍历，获得 lookahead。

- 通常需要软件/编译器注释；
- **内容导向预取（Content-Directed Prefetch）**：自动检测 load 值中看起来像有效虚拟地址的值，无需注释；
- 局限：跳转距离难以调优，指针本身存储开销大。

### 3.2.2 成对关联（Pair-Wise Correlation）

地址关联预取器本质上是一张**查找表**：触发地址 → 后续可能访问的地址。能捕捉步长关系，也能捕捉指针-目标关系。

核心挑战：
- **存储需求巨大**：需要为每个地址存储其后继，随工作集大小线性增长；
- **依赖重复**：只能预取历史上出现过的地址，无法处理从未访问过的冷地址。

### 3.2.3 Markov 预取器

最简单的成对地址关联预取器，实现为片上组关联表（以触发地址为索引）：
- 每个表项存储若干个历史后继地址，触发 miss 时全部预取（**预取宽度 width**，通常为 4）；
- 以牺牲准确率换取更高覆盖率；
- 原始方案需要约 1 MB 查找表，现代工作负载需要更大。

两大局限：
1. **Lookahead 有限**：只预测下一个 miss，触发到下一次 stall 的时间窗口太短；
2. **片上存储容量限制**覆盖率。

### 3.2.4 增大预取深度以提升 Lookahead

**方法①**：递归查表（用初始预测地址再次查表），但查表延迟高、带宽压力大，且候选地址数量随深度指数级增长；

**方法②**：在表项中存储一段连续的后继地址序列（流），避免多次查表，但需要固定最大深度，过短牺牲覆盖，过长浪费存储甚至降低准确率；

**EBCP（Epoch-Based Correlation Prefetching）**：利用乱序执行的**内存级并行性（MLP）**——每次并行 miss 组（epoch）的第一个 miss 作为触发，预取下一个 epoch 的 miss，跳过当前并行发出的地址。

### 3.2.5 通过死块预测提升 Lookahead

**死块（Dead Block）**：缓存中已存在但在被替换之前不会再被访问的块，占据存储但不再产生命中。

**死块关联预取器（DBCP）**：在预测某块成为死块时立即发出预取请求（用该块将要被替换的位置存放新数据），从而获得最大 lookahead。

死块检测方式：
- **代码关联（Code Correlation）**：识别每次进入缓存后访问该块的指令序列，以最后一次访问对应的指令作为"死亡"预测触发点；
- **时间保持（Time-Keeping）**：基于块的历史存活时长（时钟周期），预测其何时变为死块（上次存活时长 × 安全系数）。

### 3.2.6 解决片上存储限制

**方法①：标签关联（Tag Correlating）**：仅存地址 tag 而非完整地址，减少每项存储，但提升幅度有限；

**方法②：将关联表移至主存（片外）**：容量不再受限，对大工作集负载有效；代价：元数据访问延迟从几个时钟周期变为一次完整 DRAM 访问，必须通过足够大的预取深度来掩盖此延迟。

### 3.2.7 全局历史缓冲区（Global History Buffer，GHB）

**核心创新（Nesbit & Smith）**：将关联表拆分为两个结构：

| 组件 | 作用 |
|------|------|
| **历史缓冲区（History Buffer）** | 环形缓冲区，按时间顺序记录 miss 地址序列 |
| **索引表（Index Table）** | 将触发地址（或其他键）映射到历史缓冲区中的位置 |

优势：单个触发可指向**任意长度的流**，克服了 Markov 表项固定深度的限制。

GHB 分类（GHB X/Y）：
- **局部化方式（X）**：G（全局，按时间顺序）或 PC（按指令 PC 链接）；
- **关联方式（Y）**：AC（地址关联）或 DC（步长关联）。

### 3.2.8 流链接（Stream Chaining）

GHB 的局限：PC 局部化时发现的流往往很短，lookahead 不足。

**流链接**将不同 PC 产生的独立短流首尾相连，形成更长的预取序列：
- 在索引表中为每个流增加"下一个流"指针和置信度计数器；
- 利用相邻流之间的时序关联，从单次触发出发预取更多正确地址。

### 3.2.9 时序内存流（Temporal Memory Streaming，TMS）

TMS 采用 GHB 的存储组织，但将索引表和历史缓冲区**全部移至主存**：
- 可记录和回放任意长度的流，对大工作集负载有效；
- 挑战：元数据更新带来额外带宽压力（每次 miss 需更新索引表和历史缓冲区）。

**优化**：
- 历史缓冲区：片上小缓冲区合并多次连续追加为单次写入；
- 索引表：**采样更新**——只随机执行一部分索引写入，研究表明：长流在少量采样后即可被记录，频繁流会通过多次遍历被记录，覆盖率损失可忽略。

> IBM Blue Gene/Q 的 **列表预取（List Prefetching）** 是 TMS 的唯一已公开商业实现。

### 3.2.10 不规则流缓冲区（Irregular Stream Buffer，ISB）

**动机**：片外存储延迟高，限制短流的 lookahead；片外 GHB 无法支持 PC 局部化链表追踪。

**核心思路（Jain & Lin）**：引入**结构地址空间（Structural Address Space）**，仅对预取器可见：
- 随着 LLC miss 序列的推进，为每个缓存块分配连续的结构地址；
- 物理地址空间中的**时序关联** → 结构地址空间中的**空间关联**；
- 用简单的下一行流预取器在结构地址空间中发出请求；
- 两张片上表维护物理地址与结构地址的双向映射，随 TLB 填充和替换同步转移到片外。

## 3.3 空间关联预取（Spatially Correlated Prefetching）

空间关联利用数据布局的规律性：程序频繁使用具有固定内存布局的数据结构（对象、数据库记录、栈帧），访问这些结构时产生可复用的**相对偏移模式**。

**优势**：
- 学到的模式可复用于内存中许多对象，存储效率高；
- 可预取**从未访问过的冷地址**（基于布局模式计算偏移），这是地址关联预取做不到的。

**触发事件**通常是第一次对某内存区域的访问/miss，触发键必须**独立于基地址**（否则蜕化为地址关联预取）。

### 3.3.1 步长序列关联查找（Delta-Correlated Lookup）

使用连续访问之间的**步长前缀（delta 序列）**作为查找键——步长序列本身的重复性即为预测依据。

### 3.3.2 GHB PC/DC（PC 局部化步长关联变体）

**GHB PC 局部化步长关联**（GHB PC/DC）被认为是存储效率和覆盖率综合最优的预取器之一：

- 索引表键：**触发 miss 指令的 PC**；
- 历史缓冲区通过链接指针将同一 PC 的连续 miss 串联；
- 触发 miss 时，回溯历史缓冲区找出同 PC 的前两次 miss，计算两个步长（delta 对）；
- 继续向前搜索历史缓冲区，找到相同步长对的上次出现位置；
- 将匹配位置之后的步长序列应用于触发 miss 地址，生成预取地址序列。

仅需 256 项索引和历史缓冲区即可在 SPEC 基准上表现出色，但对大规模服务器/云计算工作负载的扩展性尚未得到充分研究。

### 3.3.3 代码关联查找（Code-Correlated Lookup）

**核心观察**：特定指令序列总是以相同的偏移模式访问内存——无论基地址如何变化（类似于"曲奇切模"，不同位置印出相同形状）。这种模式来源于：
- 对象方法访问对象字段的固定偏移；
- 栈帧建立时访问固定布局的栈变量。

**通用架构**（见图 3.6）：
1. 触发事件（通常是对某固定大小内存区域的首次 miss）；
2. 以触发 PC（加上区域内偏移的低位）构造查找键，查询**模式历史表（PHT）**；
3. PHT 存储对应的**空间模式（位向量）**，指示区域内哪些缓存行需要预取；
4. 用区域基地址 + 位向量计算预取地址列表。

区域大小因设计而异（从 256 B 到 8 KB 不等）。

### 3.3.4 空间足迹预测器（SFP）

Kumar & Wilkerson 的最早代码关联空间预取器，结合**解耦分区缓存**（大 sector 共享 tag，小 line 各有 valid 位）：
- 以 PC + 区域内行号为 PHT 索引；
- sector 被替换时，将触发 PC 和 valid 位向量存入 PHT；
- 目标：新 sector 分配时预取其中可能被访问的行，兼顾小 line 大小的带宽优势和大 block 大小的空间局部性。

### 3.3.5 空间模式预测器（SPP）

Chen 等人的设计，适用于常规组关联缓存，在高达 1 KB 的区域内预取 64 B 缓存行：
- 256 项无标签 PHT（不足 1 KB 总存储）可在 SPEC 上达到 95% 覆盖率、< 20% 误预取；
- 可与降低缓存泄漏能耗的电路技术结合使用。

### 3.3.6 隐形预取（Stealth Prefetching）

专为**总线广播侦听式多处理器**设计，目标是避免激进预取引发大量缓存一致性失效：
- 粗粒度跟踪共享/私有内存区域；
- 只对私有区域激进预取，并绕过代价高昂的一致性操作；
- 优势：预取操作不会在其他缓存上触发 snoop 操作。

### 3.3.7 空间内存流（Spatial Memory Streaming，SMS）

Somogyi 等人的工作，首次证明代码关联空间预取对**片外 miss（LLC miss）**和**服务器应用**有效：
- 区域大小 8 KB，针对更大内存足迹（覆盖所有片上缓存）；
- 引入**空间区域代（Spatial Region Generation）**：从首次 miss 开始，到任何一块被替换时结束；
- **过滤表（Filter Table）**：新区域代先进过滤表，第二次 miss 才转移到**累积表（Accumulation Table）**，避免单例 miss（无效区域）占据累积表存储；
- PHT 需约 64 KB；后续工作将 PHT 虚拟化存入 LLC，用小型元数据缓存加速访问。

### 3.3.8 时空内存流（Spatio-Temporal Memory Streaming，STMS）

**动机**：
- 空间预取对指针数据结构（不规则布局）效果有限；
- 地址关联预取对不频繁重复的扫描/复制模式效果有限；
- 两者互补，需要统一设计。

**方案**：整合 TMS（时序流）和 SMS（空间流）：
- 简单整合会产生突发式大量预取请求（丢失了两者各自的节流机制）；
- 关键创新：将空间模式从位向量升级为**有序偏移流（Offset Stream）**，保留 miss 的排序信息；
- TMS 提供触发序列（基地址 + PC 对），SMS 给出每次触发的空间预取地址；
- 两种流的地址通过 **delta 字段**（指示两个空间/时序流地址之间需插入的来自另一个流的地址数）交织重建总顺序，放入重建缓冲区后发出预取。

## 3.4 基于执行的预取（Execution-Based Prefetching）

不依赖 miss 序列或数据布局的规律性，而是**超前于正常指令执行**，通过实际地址计算逻辑发现预取候选地址。核心目标：比主线程跑得更快，同时忽略与内存地址计算无关的计算。

### 3.4.1 算法摘要（Algorithm Summarization）

提取并压缩数据结构遍历所需的指令序列，使摘要可以比主线程更快执行：
- Roth 等人：硬件自动识别指针 load 及其依赖链，编码为紧凑状态机，以快于正常执行的速度迭代；
- Annavaram 等人：提取程序依赖图（只保留导致 miss load 的指令子集），在专用预计算引擎中执行。

### 3.4.2 辅助线程与辅助核（Helper-Thread / Helper-Core）

利用多线程/多核处理器的**空闲上下文**运行辅助线程，与主线程并行执行，提前发出 miss 请求：
- 技术变体多样：自动/需编译器辅助、SMT 硬件/额外核、是否需要远程缓存插入机制；
- 局限：需要空闲执行资源；在高线程级并行负载中，这些资源可能已被应用线程占满。

### 3.4.3 超前执行（Run-Ahead Execution）

当核心因长延迟事件（如片外 cache miss）即将 stall 时：
1. 保存当前执行状态快照；
2. 继续取指/执行后续指令流（数据依赖于未完成 miss 的指令跳过，传播"毒令牌"）；
3. 长延迟事件完成时，从快照恢复状态，重新执行超前执行阶段覆盖的指令。

主要收益：超前执行期间发出的 load miss 起到预取效果，并预热分支预测器。

**扩展**：非阻塞流水线微架构，允许在长延迟 load 返回时仅重执行依赖指令，而非完全回滚。

### 3.4.4 上下文恢复（Context Restoration）

**背景**：虚拟化和云环境中，上下文切换会用另一个应用/虚拟机的缓存内容覆盖当前应用的缓存，造成严重的指令和数据 miss 抖动（对延迟敏感的服务应用尤为有害）。

**思路**：在上下文切换时捕获缓存内容，当该上下文恢复执行时将这些内容重新预取回缓存：
- 不需要预测预取地址，只需还原之前缓存的内容；
- 设计挑战集中在：如何关联缓存状态与特定执行上下文、管理预取时机、高效记录/存储元数据。

### 3.4.5 计算分散（Computation Spreading）

与预取正交但有相似效果：通过**线程迁移**将大型计算拆分到多个核，将相似的执行片段聚合在同一核上：
- 重复执行相似片段，使核私有的指令/数据缓存和分支预测器专门针对该类片段优化；
- 已用于分离应用代码与 OS 代码片段，以及 OLTP 线程不同执行阶段。

## 3.5 预取调制与控制（Prefetch Modulation and Control）

预取的激进程度需要动态调整，避免副作用：

| 机制 | 描述 |
|------|------|
| 污染跟踪与动态降级 | 监测预取导致的缓存污染，准确率/及时性差时自动降低激进度（Srinath 等） |
| 多核预取协调 | 协调多核各自的预取器激进度，防止带宽竞争（Ebrahimi 等） |
| DRAM bank 感知排序 | 重排预取请求以提升 DRAM bank 级并行性（Lee 等） |
| 流结束检测 | 动态检测流终止，减少短流的过度预取（Hur & Lin） |
| LRU 位置插入 | 将预取块插入 LRU 位置（而非 MRU）以控制污染，并调制发出速率（Lin 等；Wu 等） |
| 软件控制 | 程序员/编译器控制预取激进度（Verma 等） |

| 技术 | 核心思路 | Lookahead | 准确率 | 硬件开销 |
|------|---------|-----------|--------|---------|
| 步长/流预取 | 用 PC 或地址检测步长 | 少量缓存块 | 步长时较高 | < 1 KB |
| 简单地址关联 | 一个地址对应一个或多个后继地址 | 单个缓存块 | > 30% | 数 MB |
| 链式数据关联 | 自定义 FSM 追踪指针 load 依赖链 | 少量缓存块 | 特定数据结构时较高 | < 1 KB |
| 死块关联 | 关联最多两个地址和控制流到后继 | 死块事件到替换的时间 | > 50% | 数 MB |
| 时序流（TMS） | 一个地址触发一段 miss 流 | 多个缓存块 | > 50% | 片外 |
| 链式流 | 将多个地址流首尾相连 | 多个缓存块 | > 50% | 片外 |
| 不规则流缓冲区（ISB） | 地址间接层，将时序关联转为空间关联 | 多个缓存块 | > 50% | < 32 KB |
| 步长关联（GHB PC/DC） | 记录并关联步长序列 | 少量缓存块 | > 30% | < 32 KB |
| 空间流（SMS） | 将 PC 与任意 miss 偏移序列关联 | 少量缓存块 | > 50% | < 64 KB |
| 基于执行 | 辅助线程或推测超前执行 | 受执行带宽限制 | 各异 | — |

**四类预取器的本质对比**：

```
步长/流预取
    规则性强、实现简单、对指针无效
    ↓
地址关联预取（Markov → GHB → TMS → ISB）
    捕捉时序关联、适合指针结构、存储需求大、依赖历史重复
    ↓
空间关联预取（SFP → SPP → SMS → STMS）
    捕捉布局规律、存储高效、可处理冷地址
    ↓
基于执行的预取（辅助线程 / 超前执行）
    不依赖规律性、直接超前运行程序逻辑
    代价：需要额外执行资源
```
